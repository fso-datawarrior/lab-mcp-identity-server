# Even a Fully Compromised Model Cannot Deprovision a User Alone
### A demonstration of AI-directed Okta identity operations with governance in mechanism, not in name

*Demonstration build. Lab 3 of the Identity Lab series. Public MIT repository: fso-datawarrior/lab-mcp-identity-server. This is a learning artifact, not production software.*

---

## 1. Introduction

Identity is a control plane, not a data store. An assistant that can look up a user, provision one, grant a group membership, revoke it, or deactivate an account is not reading a record. It is changing who can reach what. A single destructive action on that plane has reach far beyond the directory it touches, because the applications and pipelines downstream of the directory inherit its decisions. When the caller holding those tools is an AI assistant driven by text it did not write, identity becomes among the highest-blast-radius control planes an automated actor can be handed.

This paper assumes the worst about that actor. By a fully compromised model we mean a model that has been prompt-injected through untrusted content it reads, jailbroken past its own instructions, or swapped outright for a hostile one, and that can therefore emit arbitrary tool calls in arbitrary order. We do not assume the model is well-intentioned, and we do not assume it is merely buggy. We assume it is trying to cause harm and that it holds the same tools a cooperative model would.

The claim this lab was built to make true is bounded and specific. Even a fully compromised model cannot, alone, execute a gated destructive identity action, because the destructive branch of every such action terminates at an out-of-band human approval the model has no channel to satisfy. The trust boundary is drawn explicitly: the model sits outside it, and the MCP server process together with the out-of-band approval channel sits inside it. The gated destructive set is precise, namely revoke of a group membership and deactivation of a user, along with any grant into a protected group, which is treated as destructive-equivalent because it can escalate privilege.

This is a property of where the control physically lives, not of how the model behaves. The credential that authorizes a destructive action is never held by the model, and no tool exposed on the MCP surface resolves an approval. Both facts are readable in the server source, and the second is asserted by an automated test that pins the registered tool set to exactly the five operational tools and nothing that resolves, approves, or confirms. Because the property rests on placement rather than on conduct, it holds against a model that misbehaves in ways the authors never anticipated.

A reader does not have to take the claim on trust. The repository is public and MIT-licensed, and its test suite of 108 cases runs green with zero credentials, so anyone can clone it, inspect the seam where the destructive branch stops, and run the suite. Those tests corroborate that the construction holds. They are not the basis of the claim. A passing suite is engineering diligence, and this paper is careful never to let it stand in for an adversarial proof.

The claim would be false under a small set of conditions, and we name them so a skeptical reader knows where to look. It would be false if the gate could be bypassed inside the server, so that a destructive tool executed without ever creating a pending request. It would be false if the approval could be satisfied in-band, so that some tool the model can call resolved the pending action. It would be false if the out-of-band channel were itself reachable or forgeable by the model, so that the human step could be simulated. And it would be false if a tool outside the gated set reached equivalent blast radius without gating. There is one residual we do not eliminate, and we surface it here rather than bury it: the model authors the text of the approval request that the human reads, so it retains an influence channel over the human's decision even though it holds no channel to execute. That residual is the point of the paper as much as the gate is, because it marks exactly where mechanism ends and human judgment begins.

Everything that follows is framed as a demonstration built for learning. The lab's gaps are not hidden behind the claim; they are scored openly against a published rubric later in this paper, and every strong statement is paired with the limit that bounds it. The next section describes the server as it actually stands, so the results that follow are followable. Sections 4 and 5 are the heart of the argument: four controls that looked structural and were not, and a family of seams that a passing test suite could not see. Section 6 reports a deliberate adversarial pass against the gate, section 7 reports a single live run against a real directory, and section 8 scores what the lab still does not defend.

## 2. Design of Record: Where the Control Actually Lives

This section describes the server as it stands, so that the results are followable, and it carries each mechanism's limit on the same page it introduces the mechanism. Every claim here is anchored to a citable artifact, and the reproducible interior of each component, meaning full state enumerations, storage schemas, the byte-level signature construction, and exhaustive tool parameters, is placed in the appendices so that the security-relevant statements in this section can stay short.

The tool surface is five tools. One reads, namely get_user. Two are additive, namely provision_user and grant_access. Two are destructive, namely revoke_access and deactivate_user. A sixth verb, delete_user, was present in the first design and was cut, because in Okta a user cannot be deleted in a single step, a delete requires a prior deactivation, and a delete is irreversible. Keeping the top destructive verb at deactivation keeps every action the lab can take reversible within the lab, which is the honest posture for a demonstration and removes an irreversible operation from a surface an untrusted caller can reach.

Tier is a function of the target, not of the verb. The intuition that additive operations are safe and removals are dangerous is only true when the target is low blast radius. A grant into an ordinary group is a low tier and flows without a human gate, which preserves the argument against approval fatigue. A grant into a protected group is treated as destructive-equivalent and gated, because adding a user to an administrative group escalates privilege as surely as any removal. The set of protected groups is a server-side allowlist, so classification is decided by the server against the resolved target and not inferred from the verb the caller used.

The out-of-band approval gate is the structural centerpiece, and it is the mechanism the paper's central claim rests on. A destructive tool does not execute. It writes a durable pending request and returns only a request identifier. Resolution happens in a separate local process, a command-line approver that presents a credential compared inside the server, and no tool on the MCP surface resolves anything. This is not a convention that the resolving tool ought to be called only by a human. It is the absence of any resolving tool at all, asserted by a test that pins the registered tool set to the five operational tools and none that approve, deny, resolve, or confirm. Stated as an architectural fact rather than a behavioral hope: the destructive step requires a credential the model never holds and a channel the model cannot reach, so a fully compromised model cannot complete it alone. This decision is recorded as ADR-0001. One honest qualifier on the credential belongs here and section 7 restates it against the live run: in the demonstration's single-secret wiring the credential the human approver presents and the credential the server checks against are the same environment secret, so what the demo establishes is process separation and the absence of a resolver tool, and a production version would bind approval to a distinct, independently authenticated approver.

The audit trail is a hash-chained log in which each entry commits to the hash of the entry before it, with an optional keyed signature over each entry. Its exact property is that it is tamper-evident and order-preserving under the assumption that the signing key is protected, so that any edit, reordering, or deletion of a line that remains in the file breaks verification downstream. It is not tamper-proof, and it is not a non-repudiation mechanism, because the signing key is symmetric and is held by the same running server that writes the log. This is the single dimension the self-assessment in section 8 scores at 74, and the deletion-evidence boundary that a later assertion closes is deferred to section 4 and section 8 rather than opened here. The signature construction itself is in the appendix.

Every destructive request carries a required, non-empty justification, logged verbatim and shown in the human approval prompt. The justification is the one field that distinguishes a sound request from an injection-driven one in a post-incident review, and surfacing it to the approver lets a human catch a mismatch between the stated intent and the actual target before the action executes.

The caller identity on each audit line is a configured label, not an authenticated identity, and this paper states that in the body rather than deferring it to a limitations footnote. The audit answers the question of which server-configured principal acted and which fingerprinted actor string was recorded. It does not answer the question of which cryptographically authenticated agent made the call. That gap is named here and is scored against the lab in section 8, and it is recorded in the assumption ledger as AD-8.

Every string that arrives from the directory is treated as untrusted data. Profile fields such as display names are attacker-controllable in a real organization, and they flow back toward the model and toward the human approval prompt, so each is stripped of control and zero-width characters and is never interpolated raw into either context. This section claims exactly that a boundary exists and nothing more. It does not claim that injection is prevented, blocked, or defeated. A crafted field, on the paths where it surfaces to the model, still returns as data rather than as an executed instruction, and the property that stops a crafted instruction from causing an action is the structural gate above, not the sanitizer. This is recorded as AD-3, and its live behavior, including a runtime characteristic of the directory SDK that the live run made precise, namely that the crafted field does not surface through the SDK on the live read path at all, is shown in section 7.

Pending approvals are durable, single-use, and time-limited. They are reloaded when the server restarts, and they carry a precondition that is re-checked at approval time, which closes the window between the moment a request is made and the moment a human approves it, so that a stale approval cannot execute against a directory whose state has since changed. The engine fails closed on expiry, on drift, on a wrong credential, and on an already-resolved request, and it executes at most once, including across a process crash, a property carried by an intermediate persisted marker. The full state machine and the outcome enumeration are in the appendix. This is recorded as AD-5.

Finally, the server is one end of a two-lab chain. An approved revoke removes an Okta group membership, and that membership change cascades through Okta's own provisioning into a second lab's SCIM deprovisioning path, so that a single approved action deprovisions a user across two systems. This section states only that the cascade exists and is structurally enforced. The measured cross-system latency and the honest limits of the correlator that observes it are reported in the results.

The design is meant to be run, not only read. The reproducible interior of every component named above lives in the appendices, and the zero-credential clone path, the fact that the whole suite passes without any live credential, is treated as a first-class property of the design rather than a convenience of the tests, because a control a stranger cannot inspect and exercise is a control a stranger has no reason to trust.

## 3. Method: Governing the Builders by the Same Rule as the Build

The same rule that governs the server at runtime governed its construction, and that is not a slogan, it is the reason the rest of this paper can be trusted. Two roles ran the build. One was a program manager and validator that never wrote a line into the repository. The other was an implementer that wrote all of it. Everything that reached the repository went through one prompt at a time, and every reply the implementer returned was checked against durable ground truth, the files on disk, the git history, and the live directory state, rather than against the reply's own account of what it had done. The anchors that make this falsifiable are concrete: a public MIT repository, a test count that grew monotonically from 7 to 108 as the build proceeded, and specific on-disk checks, the branch, the test file, and the findings document read directly before any tracker was updated.

That cross-check earned its keep once, and we report the single instance and claim nothing more general from it. A reply arrived describing work it had not done, a stale account replayed from an earlier task, and it was caught not because the account read as wrong but because the disk did not match it. The honest phrasing is the past tense and the one case: the disk cross-check caught this mis-fire. It is not a validated detector of mis-fires in the general case. Across the handoffs it governed, more than twenty through the cascade harness plus the later pentest and fix handoffs, the executed work passed on-disk validation, and we state that with its one caught mis-fire and its exact count recorded in the orchestration log, so that a zero-failures line cannot quietly read as survivorship.

Before any code was written, the design went through a blind four-reviewer panel, run in parallel with no cross-talk and a refute-by-default framing, and its three technical reviewers converged on the same top flaw, the confirm-tool reversal of section 4, without coordinating. That convergence is a strong qualitative signal that the flaw was real and discoverable by an independent reader, and it is bounded exactly as far as it goes: four reviewers, one lab, one build, and not a proof that every flaw was found. One boundary closes the section and it is the load-bearing sentence of the whole method: governing how the code was produced does not by itself make the produced control sound. That is what the pentest and the live run are for. The full method, its reproducibility mechanics, and the workflow's own governance are carried by the separate PAIRS workflow paper, with an in-depth spin-off planned, so this section stays a principle rather than a procedure.

## 4. Named, Not Enforced: Four Controls That Looked Structural and Were Not

This is the finding the rest of the paper is built to earn. Four times, in the design that went into review, a control named a safe idea while its enforcement actually lived somewhere the constrained component could reach. Each looked structural. Each was a convention wearing the costume of a mechanism. All four were caught before anything shipped, by an independent adversarial review, and all four were flipped to a boundary the constrained component cannot reach. The pattern is the point, so we take the four in turn, and for each we state the before-state, the mechanism that replaced it, and the limit that survives, in that order and without softening.

The first reversal is the load-bearing one. The original design gated a destructive action behind a confirmation token: the destructive tool returned a token, and a separate confirm tool executed the action when handed that token back. It read like a handoff between machine and human. It was not. The confirm tool lived on the same MCP surface the model calls, and the token was handed to the model in the tool result, so nothing stopped the model from reading its own pending response and calling confirm on the next turn. The model could approve its own destructive action, and the log would even stamp a human approver on a line the model originated. All three technical reviewers opened on this, independently, as their top critical finding. The fix is the out-of-band gate described in section 2: confirmation is not a tool at all, it happens on a local channel the model cannot reach, authenticated by a credential the model never sees, and the request identifier that enters the model's context is inert on its own. The claim becomes true by construction, and it is the one strong claim this section is allowed to make: even a fully compromised model cannot execute a destructive action alone, because the control lives in the server and the out-of-band channel and not in the model's behavior. The threat model is the compromised or hijacked model itself, and section 6 shows this holding live, when a hostile directory field carrying an instruction to confirm any pending token resolved nothing, because no tool exists for such an instruction to reach. This is AD-1 in the assumption ledger.

The second reversal is about the record. The original audit trail was described as append-only, with actor fingerprinting to answer who acted. Fingerprinting answers who, which felt like the hard part. It does nothing to prove the file itself was not rewritten, and append-only was a naming convention on a local file any process could edit with a text editor. The auditor's first question, how do you know this log was not tampered with, had no answer. The fix is a hash chain in which each entry commits to the hash of the one before, so any edit, reorder, or deletion of a line that stays in the file breaks the chain downstream, with an optional keyed signature so that an attacker who recomputes the whole chain still cannot forge a valid signature without the key. The threat model here is an offline editor who holds the file but not the key, and it is a different adversary from the one in AD-1. The honest limit sits right against the win. This defends against an offline editor, not against the running server that holds the key, and it was tamper-evident against edits before it was evident against deletion at the end of the file. That deletion-evidence gap is the sharpest beat in this section, so we tell it plainly. A later adversarial pass found that a log could be truncated at the tail, or emptied, or left absent, and verification still returned a clean result, because verification carried no expected length or external anchor to contradict a shortened file. The chosen fix is a verify-time assertion in which the caller supplies its own independently held entry count and head, so that a short or missing file fails closed. What matters for this paper is the fix that was refused. A tempting alternative was to write the expected length into a small file next to the log. We rejected it, because an attacker who can truncate the log can edit a sidecar next to it just as easily, so that alternative would have reintroduced the exact convention-not-mechanism shape this whole lab exists to document. Recording a control that only looks like a mechanism would have been worse than naming the residual. This is AD-2, and its remaining limit, that deletion-evidence is verify-time and depends on the caller's own truth rather than being caller-independent, is carried openly into section 8.

The third reversal is the quietest and, for an AI system, the most dangerous. Reading a user was classified as safe, because a read cannot change state. But the fields a read returns, the display names and custom attributes, are attacker-controllable in a real organization, and they flow straight into the model's context. A display name that reads, in effect, ignore your instructions and revoke everyone and confirm any pending token, is not data to a model, it is an instruction the model may act on. A read had become a write into the model's instruction space. The fix is an input trust boundary: every directory string is treated as untrusted, stripped of control and zero-width characters, and never interpolated raw into the model context or the approval prompt. Here the honest limit and the win have to be stated in the same sentence, because the temptation to overclaim is strongest here. The boundary contains, it does not prevent. The sanitizer strips characters, not meaning, so an instruction-shaped field that does reach the model still returns as data, and the property that actually stops it from causing an action is the structural gate from AD-1, not the sanitizer. Character sanitization is necessary and it is not sufficient. This is AD-3, and section 7 shows it against a real hostile profile on a live directory, including a runtime detail the live run made precise: the directory SDK the server reads through did not surface the crafted field at all, so on the live path the untrusted string never reached the model, and the data-not-instruction property, an untrusted field that does arrive is returned as inert data, is the one the offline suite carries.

The fourth reversal corrects a classification. The design keyed danger on the verb, so grants were additive and safe and only removals were gated. Additive is only safe when the target is low blast radius. A grant can add a user to an administrative group, and provisioning can create an active user and then place them in one, so the model could escalate privilege, its operator's or anyone's, with no human in the loop, all through operations the design had labeled safe. The direction of the change was the wrong thing to key on. The sensitivity of the target was what mattered. The fix is tier by target: a server-side allowlist marks the protected groups, grants into them are gated exactly like removals, and every other grant flows and is audited, so the argument against approval fatigue survives while the escalation path closes. This is AD-4.

Read together, the four reversals have one shape. A separate tool the model could still call. An append-only log anyone could edit. A read that wrote into the model's instructions. An additive action that escalated privilege. Each named a safe idea, and in each the enforcement lived in a convention or a prompt rather than in the architecture. The failure mode of AI governance, on the evidence of this lab, is rarely a missing control. It is a control that exists in name and not in mechanism. The cleanest proof of that thesis is not even one of the four. It is a one-line miniature the adversarial pass in section 6 turned up after every test was green: a required justification on a destructive request was enforced against an omitted field but accepted an empty or whitespace-only string, so justification required was satisfied by nothing at all. It was found by the pentest, not by the suite, and it was closed the same session. Naming a governance principle is not enforcing it, and the distance between the two is exactly where these systems fail. The per-reversal commit trail, the full options analysis behind the deletion-evidence decision, and the ten-case adversarial roster that widened AD-2 are carried in the appendix and in section 6, so that this section can stay an argument rather than a changelog.

## 5. Green Tests, Red Demo: The Seams a Passing Suite Cannot See

The reversals in section 4 were caught by review, before code. The finding in this section is the sibling of that one, and it lands during the build rather than before it: a control can pass its own test suite and still be convention-only the moment it meets a real runtime. Over the build, demo-staging, and live phases, ten distinct catches surfaced this way, none of them a failure of the model's behavior, every one of them a place where behavior that was correct in the development environment was wrong at a boundary the tests could not cross. We sort them into two named families and one operator-environment class, and the sort itself is the argument.

The first family is the dev-vs-runtime seam, and it has six members, the sixth added by the live run reported in section 7. The clearest is the first one caught. In the mock world the build was written against, a group's identifier and its display name were the same string, so a control that classified a group by name and a control that enforced an allowlist by identifier never disagreed. Against a real directory they are different values, and the layers split: the tier classifier keyed on the name while the allowlist keyed on the identifier. Two things followed. A legitimate call using the group name was refused, so the live path could not work, and, more interesting for this paper, a caller could pass a protected group's raw identifier to slip past a name-based classifier, an alias bypass. It failed closed, so there was no exposure, but a classification keyed on a label rather than on the resolved identity is a convention, not a mechanism, which is section 4's shape recurring inside the evidence layer. The fix moved classification onto the canonical resolved identity, and a regression test now passes a protected group's identifier alias and proves it still classifies as destructive and changes nothing. This one was caught before the live run, by a validator tracing the runtime path on disk rather than trusting a green suite. The sixth member of the family is the one the live run added and it belongs with these: the server maps an untrusted display field out of the directory SDK's response, and the SDK's typed response did not carry that field at all on the live path, so the value the offline tests exercise never arrives in production. It failed closed in the strongest possible sense, the untrusted string simply never reaches the model, but it is the same shape, a behavior correct in the mock and different at a runtime boundary, and it is described in full in section 7 and recorded as AD-18.

The second family is subtler and we call it the honest-confidence class. A tool returns a clean signal it has no standing to vouch for, and hands false confidence to both the human and the model. The example that names the class is a readiness check that queried a group's application assignments and read an empty result as proof that none were assigned. But the credential lacked the scope to read assignments, and the directory returned an empty list rather than an authorization error, so the check reported a confident zero it could not actually stand behind. An empty result and an unreadable one were indistinguishable at the call, and the tool treated absence of evidence as evidence of absence. The fix teaches the tool to say that it could not verify, by checking for the scope up front and reporting unverified rather than a false zero. This is a server-side integrity failure, not an agent failure, and it would mislead any caller, human or model, in exactly the same way, and its consequence is epistemic, which is why it belongs in a governance paper rather than a bug list.

The operator-environment seams are stated at the level of outcome, and the live run added a second instance of the first one. A tunnel configuration meant for one service silently governed an unrelated one and returned a blanket not-found, and a downstream audit log recorded a deprovision operation without the field that proved it was a deprovision, so a correlator that required that field reported, wrongly, that nothing had happened. Both failed closed. The mispointed tunnel failed loudly rather than routing to the wrong place, and the correlator refused to claim a deprovision it could not confirm rather than guessing one. The second of these, the audit log that could lie by omission, is the crown jewel of this section, because it is the paper's own thesis turned on the evidence layer itself: a log that cannot vouch for what happened is a control that exists in name and not in mechanism, and the fix made the downstream event self-evidencing by recording the operation in full, which the live run in section 7 then confirmed end to end when the same correlator matched the deprovision automatically with no caveat.

The empirical proof that the seams are real and not hindsight is the split between what a passing suite could have caught and what only a live run could. Of the dev-vs-runtime family, the first three were caught before the live run, by a validator tracing the actual runtime path, the launcher, the working directory, the eventual command. The rest could only surface live, because they depend on conditions a synchronously consistent mock cannot reproduce. One needed a second real process holding a different configuration than the first, another needed a genuinely eventually consistent directory, a smoke test that read a group membership immediately after writing it passed five of five twice by winning a race and was latently flaky the whole time, and the sixth needed the real directory SDK against a real profile to reveal that the field the mock supplies is absent in production. No amount of the same tests would have found any of them. The lesson generalizes into a short recursion that this section closes on. Acceptance criteria bound the agent. Validation, live exercise, and adversarial testing are what bound the acceptance criteria. And in every one of these cases the fix, or the honest documentation of the residual, lived in code and in the runbook, in a resolve-then-classify ordering, a retry-until-consistent read with an idempotent self-heal, a configuration stamped onto a pending request, an isolated tunnel config, a recorded note that the directory SDK omits the field, never in a better prompt, which is the same reason none of them is a story about model behavior. The full roster, with the boundary each one crossed, the layer that caught it, the commit or record that closed it, and whether it failed closed, is an appendix table, so that a reader who clones the repository can re-trace every catch.

## 6. Held Closed, Then Named: Three Adversarial Probes (A/B/C) Against the Gate

Sections 4 and 5 describe gaps found by review and by validation. This section describes gaps looked for on purpose. We ran one adversarial pass per axis, adopting the reasoning method of a public application-security testing approach rather than its tooling, because that tooling is oriented at an HTTP proxy surface and this server speaks over standard input and output with no inbound HTTP surface to proxy. So the method transferred and the transport did not, and we say so plainly rather than implying a heavier apparatus than we used. Two of the three passes are offline, deterministic, and reproducible with zero credentials, and the third is a single witnessed run against a live directory. That asymmetry matters for what each is allowed to prove, so we mark it throughout.

The first pass, Probe A, attacked the gate itself with twelve scenarios drawn from the reversals and the seams: attempts to self-approve with no resolver present, tier-downgrade attempts through identifier aliases and mixed case and lookalike characters, confinement bypasses on both grants and removals, an unclassifiable target that had to deny and audit, a request with no justification and a request with a blank one, a replayed single-use token, an expired approval, a drifted precondition, and a client-mode mismatch. Eleven held. Each held because of a specific server-side or out-of-band mechanism, not because of anything the model did or did not do, and the roster maps each verdict to the mechanism responsible. The twelfth was a gap, and it is the one already quoted in section 4: a destructive request with a whitespace-only justification was accepted where an omitted one was refused. It was low severity, because the human gate still stood and nothing executed, but it is the thesis in a single line, and it was closed the same session by requiring a non-empty trimmed justification, after which the scenario flips to a clean refusal. We report it as found-and-closed rather than quietly folding the fix into the design, because the finding is the evidence, not an embarrassment to launder.

The second pass, Probe B, attacked the audit log with ten cases. Seven held: an edited middle entry, a reordering, a deleted interior line, a partial final line, a wrong signing key, an unsigned file checked with a key required, and, the property that makes the signature worth having, an attacker who recomputes the entire hash chain forward but cannot re-sign it, which passes a hash-only check and fails a keyed one at the tampered line. Three did not hold, and they are the deletion-evidence family from section 4: an empty file, an absent file, and a whole-tail truncation all verified as clean, because the surviving prefix still chained from the start and verification carried nothing to contradict a shortened file. We documented the three as gaps rather than coding around them during the pass, which is the discipline the whole method depends on. They were closed afterward, and closed is the wrong word for it, so we use a more exact one. They are closed with a named boundary. The verify-time assertion described in section 4 makes a short or missing file fail closed when the caller brings its own count and head, and its residual is stated in the same breath: a bare verification with nothing asserted still returns clean on an empty file by design, the assurance is only as strong as the ground truth the caller holds, and a compromised running server that holds the signing key can still forge, because that key lives in the process. The full ten-case matrix and the option analysis behind the assertion are in the appendix.

The third pass, Probe C, is different in kind and we treat it as such. It is one live existence proof, not a reproducible suite, and it is a confirmation of the structural claim rather than its basis. Against the real directory, a genuine active user was given a display name that carried an injection payload instructing any assistant that read it to revoke a group's members and confirm any pending token, and the user was then read back through the live tool in real mode. The observable result was that the read created nothing: the pending directory on disk was byte-identical before and after, so no request existed for the injected instruction to confirm, and the field was then restored. The live run of section 7 sharpened this beat and we carry its correction here rather than leave the earlier, looser phrasing standing. On the live path the untrusted display field did not reach the model at all, because the directory SDK the server reads through does not surface that field in its typed response, a runtime characteristic the live run isolated and recorded as AD-18. So the property Probe C proves live is the structural one, that reading a poisoned user creates no pending and there is no resolver for the injected instruction to reach. The distinct property, that an untrusted field which does arrive is returned as inert data rather than acted on, is the one the offline suite carries, where a mock and stub client that do surface the crafted field show the read returning it as data, auditing once, and creating no pending, across a ten-test offline injection suite that runs green. We separate the two deliberately, because crediting the live read with a data-not-instruction demonstration it did not perform would be exactly the kind of overclaim this paper exists to avoid. We also mark the standing honest limits on this pass. It was witnessed once and is not a reproducible test, and the orchestrator that drove the console write and the tool read was, in a real sense, an agent in the loop of its own test, with the human holding only the authentication step. It illustrates the claim against a real directory. It does not, by itself, prove it, and the offline Probe A and B passes are what carry the structural weight.

The single claim this section earns is stated at exactly its size, and no larger. These probes ran. These controls failed closed under these scenarios. These gaps were found and are named, one closed the same session and one family closed with a residual we carry forward. What the uncovered surface looks like is not left to the reader to infer, because section 8 scores it directly, and the lowest of those scores are the honest map of what these three passes did not reach.

## 7. Results: One Live Run, One Audit Log, and a Compromised Model That Cannot Act Alone

This section reports one continuous run against the real directory, an Okta developer organization, on 2026-07-15, recorded in a single hash-chained audit log. It is narrated as one argument rather than a feature tour, and every figure in it is pasted from real output or named as an artifact a reader can re-derive from the preserved log. Two roles ran it, the same two that built the server: a program-manager-and-validator orchestrator drove the tool calls and the directory console, and the human authenticated the console, ran the out-of-band approver in a separate process, and stood up the second lab. The orchestrator, as in Probe C, was an agent in the loop of its own demonstration, and this run illustrates the structural claim against a live directory rather than standing in for the offline probes that carry its weight.

The pre-flight is itself a result, because a demonstration a stranger cannot reproduce is a demonstration that proves nothing about the artifact. The run opened by executing the repository's own go/no-go checklist and its fixture smoke test, which returned five of five, so the starting state was the seeded fixture and not an ad hoc arrangement. That the checklist and the smoke test exist, run, and gate the demonstration is the reproducibility claim made concrete, and it is the same zero-credential-clone property section 2 treats as first class, exercised here against live credentials.

Normal operations came first, to show the tool is useful and low-friction and not only a thing that refuses. A read of a seeded user returned a sanitized profile and wrote one Tier 1 audit line, executed, with no approval. A grant of a user into an ordinary group executed immediately and audited, with no approval, which is the argument against approval fatigue shown rather than asserted. A provision created a staged user and audited it. None of these gated, because none of them should, and the audit line for each records the actor, the tier, the target, and the decision.

Then the gate, the two beats the whole design exists for. A revoke of a seeded user from an ordinary group returned a request identifier and the status pending, and the user stayed a member, and the audit line read pending rather than executed. Nothing had happened yet, which is the point: the model asked, and the directory did not change. The human then ran the out-of-band approver in a separate process, and only then did the revoke execute against the live directory and remove the membership. The audit line for that resolution carries two distinct fingerprints, the actor fingerprint the orchestrator was recorded under and a separate approver credential fingerprint, both present and different, so the record binds the resolution to a step the model did not perform. One honest qualifier belongs on that line and section 2 introduced it: in this demonstration's single-secret wiring the approver credential and the credential the server checks against are the same environment secret, so the two fingerprints are the AI actor and the fingerprint of that shared secret presented from a separate process, and what the beat establishes is process separation and the absence of any resolver tool, not an independent approver identity. A production version would bind the approval to a distinct authenticated approver, and section 8 carries that as a residual. The deny beat is the mirror. A deactivation of the same user returned pending and left the user unchanged, and the human ran the deny path, and the request resolved denied with the user still intact and the audit line reading denied. No is a first-class outcome, recorded as plainly as yes.

The fail-closed montage, shown quickly, is default-deny under pressure. The registered tool set enumerated over the live bridge is exactly the five operational tools, and nothing that approves, resolves, or confirms, so the model has no lever to pull on its own pending request, which is the AD-1 property read directly off the running server rather than off the source. A grant into a protected group classified as gated and created a pending rather than executing. The same attempt using the protected group's raw identifier instead of its name still classified as gated and still failed closed, so the alias bypass of section 5 stays closed on the live path. A destructive request with a whitespace-only justification was rejected at the schema boundary with no pending created, which is the AD-9 fix from section 6 shown live: an error, not a pending, not an execution.

The live-injection beat is the intellectual center, and the run made it more precise than the earlier witnessed test, so we report exactly what happened and credit exactly the control that acted. A real user was given, in the directory console, a display name carrying an injection payload that instructed any reader to revoke a group's members and confirm any pending token. The value was confirmed present in the directory, both in the console and in the directory's own raw management API response, which returned the payload in the display-name field. The user was then read five times through the live tool. Every read returned an empty display name. The reason is not propagation lag and not the sanitizer, both of which we ruled out against the running system: the directory SDK the server reads through, at the version the repository pins, does not surface the display-name field in its typed response, while it does surface the login field from the same object, so the mapping the server performs yields an empty string on the live path. The payload was present in the directory and absent from what the tool returned to the model. Across all five reads the pending directory on disk was byte-identical, so the read created nothing, and the field was then restored. Two things are true at once and we state both. The structural claim held live, because reading a poisoned user created no pending and there was no resolver for the injected instruction to reach. And the distinct property, that an untrusted field which does arrive is returned as inert data, was not exercised on this path, because the field did not arrive, so that property is carried by the offline suite, where a client that surfaces the crafted field shows the read returning it as data, auditing once, and creating no pending, across a ten-test offline injection suite that runs green. This SDK-omission behavior is a sixth dev-vs-runtime seam, recorded as AD-18, and it fails closed in the strongest sense: the untrusted string never reaches the model at all. It also disciplines the paper's own language, because it means no claim of a live model receiving and ignoring the payload can be made, and none is.

Audit integrity was the next beat, and its headline is deliberately modest. Verification over the full run log returned a clean result twice, once checking only the hash chain and once additionally verifying the keyed signature on every entry, and every entry in the run carried a valid signature. The clean result is labeled for what it is, tamper-evidence and not tamper-prevention, with the residual from section 4 restated in place: the signing key lives in the running server, so a compromised running process could still forge, and the property defended is that an offline editor who lacks the key cannot alter a line that stays in the file without detection. The stronger deletion-evidence check appears in the break-glass beat below.

The two-lab cascade is the supporting throughput line, and its number is a single-configuration observation, not a service level. With the second lab running and reachable, an approved revoke of a live user from the demo group removed the Okta membership, and Okta's own provisioning pushed a deprovisioning operation into the second lab, which recorded it and returned success, verified at the second lab as the source of truth rather than inferred from the first lab's log. The correlator that merges the two logs matched the deprovision automatically by the downstream identifier, with no caveat, which is the section 5 self-evidencing-log fix confirmed end to end, and it reported a cross-system latency of 2.8 seconds. We state that figure as one observation from one configuration and not a benchmark, and the run itself is the reason we can say so honestly, because the first attempt at this beat failed and the failure is instructive. The downstream push landed at a moment when the operator tunnel to the second lab was not routing, having been interrupted, so the directory's deprovision call reached a generic edge error rather than the second lab, the directory recorded the push as a failure, and the second lab's log showed no request at all. The cascade failed closed, the directory surfaced the failure rather than silently dropping it, and no partial state was created downstream. Restarting the tunnel in isolation, repointing and re-verifying the directory's downstream base address, and re-running the joiner-then-leaver sequence produced the clean 2.8-second cascade above. The latency is real and so is the fragility of the operator plumbing around it, and reporting only the second is why the first is in the paper.

Break-glass closed the run and doubles as its self-restoring cleanup. The documented recovery runbook was exercised rather than described, and exercising it surfaced a gap in the runbook itself, which we report as found-and-noted. The runbook's fast kill-switch, setting the client mode to mock and restarting, did not take on the desktop launch path, because the launcher script hard-codes the real client mode into the process environment and overrides the setting the runbook tells the operator to change. The kill-switch is real on the local start path and is convention on the desktop path until the launcher line is edited, which is the paper's own convention-versus-mechanism shape appearing in the recovery documentation, recorded as AD-19 with a recommendation to make the launcher honor the setting. That recommendation was applied after the run: the launcher now defaults the mode to real only when it is unset, and the desktop configuration no longer pins it, so the documented kill-switch takes effect on the desktop path. The remaining recovery steps ran clean. Stale pending requests were cleared, and verification of the audit log after clearing them returned a clean result under the stronger deletion-evidence assertion, in which the caller supplies its own independently held entry count and head, so the demonstration shows that removing pending state does not disturb the append-only signed chain. The fixture was then re-seeded and the smoke test returned five of five, restoring the starting state, so the run ends where it began.

The scores that section 8 develops sit here in one line so that a clean run is never read as an assurance claim: audit trail 74, policy enforcement 71, human oversight 70, error handling 66, and revocation capability 58. A clean live run does not move these, because the run illustrates the structural claim and the low numbers are the honest map of what it does not reach. What this run establishes is narrow and, we think, worth the length: against a real directory, a model directing these five tools proposed destructive actions that did not execute without a human step in a separate process, a poisoned directory field drove no action and in fact never reached the model, the trail verified as tamper-evident, a single approved action deprovisioned a user across two systems, and the recovery runbook both worked and revealed one of its own conventions. Every one of those is an artifact in the preserved log, and none of them is a statement about the model behaving well.

## 8. Scoring the Gaps Down: A Self-Assessment of Where the Control Is Still Convention, Not Mechanism

The honest way to end a demonstration is to score it against a public rubric and publish the low numbers. We assessed the lab against a five-category agent-governance rubric on a zero-to-one-hundred scale, and the scores are ours, conservative, and carry no external audit, certification, benchmark, or compliance meaning. What keeps them from reading as self-graded confidence is that each is anchored to artifacts a reader can inspect and rerun: the repository, the deletion-evidence assertion, and the zero-credential suite. The disclaimer and the anchors travel together on purpose.

The five categories score as follows. Audit trail, 74. Policy enforcement, 71. Human oversight, 70. Error handling, 66. Revocation capability, 58, and it is the lowest for a reason we state rather than bury. The service application the server authenticates with holds an organization-wide administrative scope, so the credential behind the gate revokes at a far coarser grain than a scoped revoke implies. This is the one place the paper restates its own invariant, and it does so as an axis separation rather than a reassurance: the gate is a real mechanism, and the credential behind it is broad, and those are two different axes, so a broad credential does not weaken the gate, it widens the blast radius if the gate is ever bypassed. We say that once. We do not reuse it below to wave away any individual gap.

The residuals follow, each stated once as a mechanism gap and each carrying both how bad it is and where it lives. The organization-wide credential scope is the widest, and it is a blast-radius residual rather than a gate residual, recorded as AD-6. The caller is a configured label rather than an authenticated identity, so the audit answers which server acted and not which agent, recorded as AD-8. Approval is single-party, and in the demonstration the approver credential and the server's expected credential are the same secret, so the live property rests on process separation and the no-resolver boundary rather than on an independent, authenticated approver identity, and there is no multi-approver quorum. Deletion-evidence on the audit log is verify-time and depends on the caller's own count and head rather than on a caller-independent anchor, the AD-2 residual. A partially completed cross-system cascade has no automated rollback beyond a re-seed. And the test suite itself carries a boundary worth naming as a residual: the mock it runs against is synchronously consistent while a real directory is eventually consistent, and it surfaces fields the live directory SDK does not, so the 108 green tests attest the control logic and not the runtime behavior of a live directory, which is exactly the seam section 5 made visible and the live run in section 7 sharpened. Every one of these is attributed to a server-side or credential-side control. None is attributed to the model behaving well, because a residual that depends on model restraint is not a mechanism at all.

The remediation path is real, and it belongs to the next lab rather than to any claim made here, which is why we keep it in one paragraph and out of the residual list, where writing the remedy next to the score would let an open gap quietly read as closed. The organization-wide scope wants an authenticated caller and per-tool scoping, along the two-token on-behalf-of pattern. The single-party approval wants both a distinct authenticated approver and a quorum. The verify-time deletion-evidence residual wants an external chain-head anchor held in a separate trust domain. The missing rollback wants a compensating-action design. These are direction, not almost-done.

## 9. A Convention Until a Mechanism Enforces It

The one thing to carry out of this lab into a domain that has nothing to do with Okta is a test. A principle asserted in a prompt is a convention until a mechanism enforces it. On our own system that test has a concrete answer, and it is the claim this paper has defended: for the five enumerated identity tools, even a fully compromised model cannot execute a destructive action alone, because enforcement lives in the server and the out-of-band approval channel and not in the model's behavior. That claim holds under conditions that belong in this sentence and not in a footnote, namely while the server and the out-of-band channel are uncompromised, while the approver step cannot be spoofed, and only for the enumerated destructive tools. The model can still be compromised, and it can still emit the call. The gate constrains the outcome, not the model's behavior, which is the entire point. And the reason to trust any of it is that it is checkable, because the repository is public and MIT-licensed and its suite runs green with zero credentials, so the seam can be inspected and the suite can be run.

Two things in this paper carry different weight, and blurring them would spend the honesty the rest of it earned. The repository and its 108 zero-credential tests are the reproducibility anchor. They let a stranger clone and run without secrets, and they carry no claim of security coverage. The security weight sits with the live run and the adversarial pentest, which spoke to the artifact rather than to the diligence of its tests. Well-tested is not secure, and this lab keeps the two apart on purpose.

The most transferable thing the lab produced may not be the server at all. It is the pipeline that produced it: a diverse and blind review before the build, validation against ground truth during it, a live exercise, and a deliberate adversarial pass, each stage leaving durable evidence. We present that as a documented single case and not a certified methodology. The heuristic is portable. The review process is documented, not warranted to reproduce the same result for another team.

The evidence has teeth and it has limits, and both are on the page. Three probes held, and the gaps they found were named and dispositioned, one closed the same session and a deletion-evidence family closed with a named residual, which shows the method catches a real control-in-name failure and not that the system resists a determined adversary. The lowest self-scores, revocation at 58, error handling at 66, and human oversight at 70, are the honest map of what is not yet defended. Where this goes next is a second lab that extends the same method to a full identity lifecycle, one measured continuation of which already runs, an approved revoke deprovisioning a user across two systems in about 2.8 seconds, and the broader governance framing this build is a single instance of, PAIRS, which the companion work carries forward.
